# v0.1.11a9

**If you installed v0.1.11a8 and used `--watch-aws-tree`, please update
to this version.**

## The problem

A flag added in v0.1.11a8, `--watch-aws-tree`, wildcard-subscribed to
the entire reserved `$aws/things/{blid}/#` namespace. A field tester
(chairstacker) tried it — the run hung after sending the mission-start
command, needing Ctrl+C to exit. Worse: a completely separate process
run afterward, `roombapy-prime-verify-named-shadows` — previously
reliable against this exact account — then failed all four
named-shadow requests with timeouts, including one (`rw-settings`)
that had never failed before.

AWS IoT's own documentation explains why. Topics starting with `$` are
reserved, and "unsupported publish or subscribe operations to reserved
topics can result in a terminated connection." The Device Shadow docs
are more specific still: "we recommend that you avoid wild card
subscriptions to shadow topics... avoid subscribing to topic filters
like `$aws/things/thingName/shadow/#`" — exactly the kind of pattern
this flag used, one level up. The most likely explanation is that AWS
IoT terminated or degraded the connection in response — not a purely
local hang, since a second, independent process was also affected.

## The fix

`--watch-aws-tree` has been removed entirely, not just discouraged.
Passing it now raises a clear error instead of silently doing
something risky.

`--watch-shadow-delta` — the other new flag from a8 — is unaffected
and remains safe to use. It was never a wildcard; it subscribes to
exactly one specific topic, `$aws/things/{blid}/shadow/update/delta`,
the same path AWS's own documentation uses as the example in its IAM
policy guide for this exact feature.

## Also in this release

Unrelated to the above, carried over from work already in progress:

- **`trigger_echo_via_shadow()`** — a new, still-experimental
  hypothesis for the "find my robot" chime feature, prompted by a
  separate bug report (the existing REST-based locate action doesn't
  actually work). Not yet confirmed against a real device.
- **Discovery response caching** — cuts one of the two currently-
  redundant login round-trips during `ha_roomba_plus`'s first-time
  onboarding, with no code changes needed on that side.
- **`PrimeFactory.create_prime_robot()`** gained an optional
  `login_result=` parameter, letting a caller skip a redundant login
  when it already has a fresh one — the mechanism the discovery-cache
  improvement above builds on.

450/450 tests green, ruff clean.
