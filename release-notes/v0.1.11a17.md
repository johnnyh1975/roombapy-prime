# v0.1.11a17

The first live confirmations of two staged write-test scripts, plus a concrete next test for
region commands based directly on what a real device test revealed.

## 🎉 First live confirmations: schedule and favorite writes work

**`verify-schedule-write` stages 1 and 2 CONFIRMED WORKING LIVE** (chairstacker): resending a
household's schedules unchanged, and disabling one specific schedule, both genuinely took
effect. Real-world note, not a bug: the real app's own Automations screen doesn't always refresh
in real time after a write — `get_schedules()` itself reflects the change immediately regardless
of what the app's own screen shows at that moment.

**`verify-favorite-write` stages 1 and 2 CONFIRMED WORKING LIVE** (chairstacker) too. Two real
bugs found and fixed along the way:

- A hardcoded test-favorite name caused a genuine HTTP 409 conflict on a second stage-3 run
  (the name now includes a timestamp, so every run is unique)
- **Confirmed caveat**: a favorite created with empty `command_defs` is real and listable via
  `get_favorites()`, but was **not visible in the real app's own UI at all**. A new standalone
  `--delete FAVORITE_ID` command was added specifically for this — cleans up a favorite by ID
  directly, no app visibility required

## ✨ New: region-commands stage 1b, from a real negative result

Stage 1's own first live test (chairstacker) produced no observable effect, and the actual
payload sent had no `"initiator"` field at all — the stored favorite's own `command_def` had
`initiator=None`, which `RoutineCommand.to_json()` correctly omits when unset. This matters
because the original hypothesis behind this whole transport was that `"command"` AND
`"initiator"` are shared keys with the confirmed-working simple-command payload — the real test
accidentally exercised a version missing that second field.

`--send-with-initiator` tests the natural next, still-minimal step: identical to stage 1 in
every other way, with only `initiator="localApp"` added if unset — purely additive, nothing
overridden.

## 🐛 Fixed: `verify-region-commands` stage 2 crashed against EVERY real favorite

`TypeError: replace() should be called on dataclass instances` (jayjay, real device test).
Favorites are always constructed with their `command_defs[].params` kept as a plain dict, by
design — never upgraded to a `CommandParams` instance. `_build_modified_command()` assumed the
latter unconditionally; now branches on the actual runtime type instead, and correctly preserves
any other fields already present (e.g. a favorite's own `cleaning_profile`/"profile") while only
changing suction level.

## ✨ New building blocks for room-name resolution

- **`PrimeRobot.get_household_id()`** — finds the household_id containing a specific robot,
  handling both possible `get_user_households()` response shapes defensively.
- **`build_room_name_map(map_versions, blid=None)`** (`models/robot_info.py`) — generic
  `{room_id: name}` lookup from a list of `P2MapVersion` objects. Groundwork for calendar/schedule
  features that need to show a real room name rather than a bare region_id.
