# v0.1.11a12

An urgent hardening release, prompted directly by a real field
report: an integration that lost MQTT connectivity and stayed lost
across multiple full application restarts.

## Two related reconnect weaknesses fixed

1. **`reconnect()` reused the same token, even if it had already
   expired.** If a disconnect happened after the token's ~1h
   lifetime ran out (or the proactive refresh task below had died),
   every reconnect attempt would keep failing forever, retrying at an
   ever-increasing backoff but never able to succeed. Fixed: when a
   `relogin` callback is configured (the default via `PrimeFactory`
   with `auto_refresh=True`), every reconnect now fetches a fresh
   token first instead of blindly reusing the existing one.

2. **The proactive background token-refresh task had no error
   handling at all.** A single failed refresh attempt — a transient
   network blip at exactly the wrong moment — would silently kill
   this task for good, with no further proactive refresh ever
   happening again and no log line anywhere pointing at it. Fixed: a
   failed attempt is now logged and retried after a short delay,
   rather than ending the task permanently.

Together, these close the specific "stuck forever, no visible error"
failure mode — regardless of whether this turns out to be the exact
mechanism behind any specific report, both are real, independently
justified fixes.

468/468 tests green, ruff clean.
