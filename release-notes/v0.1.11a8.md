# v0.1.11a8

The battery/charging status investigation continues — no answer yet,
but a real correction to our own reasoning, a new topic family, and a
partially-resolved side question about live-map rendering.

## Named shadows: a hypothesis tested and disproven

Three named shadows this library had never queried before —
`rw-constatus`, `rw-schedule`, `rw-software` — are now fully
confirmed live (chairstacker). `rw-constatus` was the strongest
candidate for battery/charging, reasoned from a native-app symbol
trace. It turned out to be MQTT/AWS-IoT connection status instead —
the name's surface resemblance to "connection status" was accurate,
but pointed at the wrong kind of connection. New
`ConnectionStatusShadow`/`ScheduleShadow`/`SoftwareStatusShadow`
models capture all three. All five named shadows this investigation
opened up are now fully enumerated — none contain battery data.

New `get_named_shadow(name)` on `PrimeRobot` (a general form of what
`get_state()`/`get_settings()` were always thin wrappers around) and
a new script, `roombapy-prime-verify-named-shadows`, exist to check
any named shadow directly. The same three checks now also run
automatically as part of `roombapy-prime-validate`.

## A new topic family, found in the same capture

`dock/paddry/report` — never seen before — fired right as a mission
started. Lifetime dock/pad-dry counters, not battery data itself, but
the topic name suggests a family shaped like `dock/{reportType}/report`.
If a sibling report type for charging exists, this is the most
concrete lead so far. New `DockPadDryReport` model.

`mission/timeline/request` was also confirmed live for the first
time, not just from decompiled native symbols — the standalone form
of a field already modeled since a6.

## A real mistake in our own prior claims, corrected

A separate, parallel investigation caught something: this project had
claimed in three places that live mission status "does not flow
through get_state()/watch_state() at all." The actual evidence only
ever covered `get_state()` — two point-in-time snapshots, compared.
`watch_state()`'s own persistent push subscription had never been run
live during an active mission. That's now corrected, and two new
flags on `roombapy-prime-verify-mission-timeline` — `--watch-shadow-delta`
and `--watch-aws-tree` — exist to actually test it. Every wildcard
capture so far has also only ever covered the robot's own topic tree,
never the broader `$aws/` namespace where AWS IoT's own shadow
mechanism lives.

## A security fix, prompted by a real leak

More than one tester pasted raw terminal output containing full
presigned S3 URLs with live AWS access tokens intact. Neither existing
redaction path caught this. New `redact_aws_url_secrets()` strips just
the secret-bearing query parameters, applied both to `--dump-config`
output and directly at print time in every script that shows a raw
payload.

## Live-map rendering: rawmap is not a ready-made image

Separately from the battery investigation: `livemap_url_raw`
("rawmap") is confirmed to be zlib-compressed data that decompresses
to something not recognized as any image format. Rules out the
simplest path for a future live-map camera/image entity. Investigated
without ever seeing the actual map content — a tester ran byte-level
statistics and rendered candidate images locally, on their own
machine, checking the results themselves rather than sharing their
home's actual layout.

443/443 tests green, ruff clean.
