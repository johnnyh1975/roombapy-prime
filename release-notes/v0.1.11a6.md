# v0.1.11a6

A second, longer live capture — this time watching all the way through
stop and dock, with a wildcard subscription running the whole time —
turned up the answer to an actively open question, plus a real bug in
the tooling that was almost missed because of it.

## Position data: answered, and simpler than expected

The open question from the last release was "does position/pose data
flow over MQTT, and how do we ask for it." The answer turned out to be
simpler than the question assumed: `"pos_update"` messages carrying live
position/path data arrived repeatedly and unprompted throughout the
mission — no request needed at all. The exact topic isn't nailed down
yet (see below for why), but the data itself is confirmed real.
`update_expire_ts` turned out to stay fixed across several consecutive
messages rather than counting down per-message — consistent with a
renewable ~60-second streaming session, not a per-message expiry (an
earlier note here got this detail wrong; corrected after actually
checking the numbers).

A few smaller things came out of the same capture: a new optional
`timeline_request_id` field on report messages, and a refined (not yet
fully confirmed) read on `RoomEvent`: `area` looks like the room's
target size, `total_area` how much was actually covered on this visit.

## A live map feed, and it's already usable

The same capture showed `map_update` push notifications arriving
repeatedly throughout the mission, each carrying a presigned URL to the
current live map — in the exact same `p2mapv_geojson.tgz` format this
library already knows how to download and parse. New
`models.LiveMapUpdate` captures the shape; no new download or parsing
code was needed, since `download_map_bundle()`/`parse_map_bundle()`
already handle this format for REST-fetched bundles. The robot's own
matching upload side (`uploadP2MapLive`/`uploadP2MapMission` and a
one-time `NEW_P2MAP_AVAILABLE` notification after the mission ended) is
documented alongside it, though not modeled — this library only ever
observes that side, never constructs it.

## `"fin"` and `"pause"`: two more event types, live

Both previously confirmed only via the historical mission-history
endpoint, now also confirmed on the live `mission/timeline/report`
channel itself. `"fin"` marks a mission as concluded. `"pause"` turned
out to be what `send_simple_command("stop")` itself produces in the
timeline — there's still no confirmed event type that corresponds to
"stop" by that name.

## A real bug, found almost by accident

The same capture that found `pos_update` had 81 messages on the wildcard
channel — and all 81 printed under one identical label, with no way to
tell afterward which of them arrived on which actual topic.
`_watch_one()` was printing the static watch label instead of each
message's own `response.topic`. Invisible for a specific-topic watch
(the two are the same there), but silently discarded exactly the
information a wildcard capture exists to reveal. Fixed — now prints and
stores the real topic per message.

Also confirmed clean this session: a full structural re-check of a
second live map bundle found `ManifestFeature`, `PolicyZoneFeature`, and
`BorderFeature` all matching their existing models exactly. One
genuinely new field turned up on the room side: `RoomFeatureProperties`
now includes `visibility`, confirmed from the bundle's structure but not
previously known.

423/423 tests green, ruff clean.

## Known limitations

Battery/charging/docked status (`RobotStatusV2`) remains completely
unconfirmed — even the full capture through stop, dock, and 60s of
post-dock watching found nothing. `pos_update`'s and `map_update`'s
exact topics are still unknown — the capture that found them predates
this release's own tooling fix, so a re-run is needed to settle them.
`cur_path`'s internal shape is rigorously verified structurally (29/29
messages checked, zero exceptions) but the numbers' actual meaning is
still a hypothesis, not confirmed against any decompiled source. Only
one of the nine V1 map-edit command types (`SetRoomMetadataV1`, room
renaming) has ever been live-tested — the other eight remain
decompilation-confirmed only.

## Tooling improvements in this release

- `_watch_one()` now prints a distinct-topic frequency summary once a
  watch ends, alongside the response-topic fix above — a wildcard
  capture with dozens of messages is still tedious to read one at a
  time; the summary shows which topics were active and how often
  before diving into any of them.
- `--post-dock-watch-seconds`'s help text now says explicitly: `"fin"`
  fires within the same second as the stop command, not after the
  robot physically reaches its dock — the default (30s) is unlikely to
  be long enough to catch battery/charging status specifically; a
  much longer value is recommended for that investigation.
- `--dump-config`'s saved JSON now also gets a topic-grouped sibling
  view for every watch entry, matching what the terminal already
  showed — previously only the terminal output was grouped, the saved
  file stayed a flat list.
