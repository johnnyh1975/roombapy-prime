# v0.1.11a7

A third independent tester (jayjay13011) ran the updated tooling from
a6 â€” and the topic-tracking fix from that release immediately paid off:
this is the first capture that shows exactly which MQTT topic each
message arrived on, settling a question that's been open for two
releases.

## The exact live-map topics: confirmed

`{prefix}/things/{blid}/livemap/update` carries both live position
updates and map-ready notifications, discriminated by which key is
present in the payload. This is exactly the pattern `livemap_topic()`
and `watch_live_map()` already used â€” both existed, fully built, from
an earlier session, but had never been confirmed against a real
device. They have been now. No code change was needed for the topic
itself, only updating the docstrings from "untested analogy" to
"confirmed."

The same capture's full topic breakdown identified 7 distinct topics
in total: `livemap/update`, `filexfer_req`, `filexfer_resp`,
`livemap/cmd`, `mission/timeline/report`, `cmd`, and `service_event`.

## A real value, not just a marker

`PositionUpdateMessage`'s `operating_modes` field was documented as a
possibly-fixed marker whose meaning was unconfirmed. It isn't fixed:
the same capture showed it as `0` for the first few seconds of a
cleaning mission, then switching to `5` for the rest of the observed
period. This settles an open tension between two competing readings of
`cur_path`'s wire format in favor of the simpler one already
implemented.

## Battery status: stronger negative evidence, still not found

This capture watched a full 300 seconds after sending stop and dock â€”
specifically to give the robot time to actually get there â€” with every
one of the 7 topics identified by name. None of them carried anything
battery or charging related. This doesn't prove the data is
unreachable over MQTT, but it does rule out two explanations that were
still on the table: not watching long enough, or missing it mixed into
another topic's payload.

## A value that doesn't quite always match

An earlier note claimed `xferId` exactly equals the Unix timestamp
encoded in its message's own `p2mapv_id` field, based on a handful of
examples. Checked against 17 examples from this capture instead: 16
matched exactly, one was off by one second. Still clearly the same
underlying value, just not a guaranteed exact match every time â€”
corrected to reflect that.

## Cleanup

A `LiveMapUpdate` model was added in a6 before this session realized
equivalent, better-evidenced models (`MapUpdateMessage`,
`PositionUpdateMessage`) already existed elsewhere in the codebase,
just unconfirmed. Removed in favor of enhancing those instead of
maintaining two overlapping versions â€” `MapUpdateMessage` now carries
the two fields the removed class had added (`livemap_url_raw`, an
outer `timestamp`).
