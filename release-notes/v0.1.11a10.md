# v0.1.11a10

Three genuinely new leads this release, none confirmed against a real
device yet — this version exists to make them testable, not to claim
they're solved.

## The strongest battery lead yet: four shadows we never knew existed

Every named shadow this project had ever checked came from the app's
own command config — which only lists things you can *write* to nothing
writes to a read-only shadow, so four of them were invisible for
exactly that reason. `MQTTTopics.java` reveals them directly:
`ro-currentstate`, `ro-stats`, `ro-services`, `ro-configinfo`.

`ro-currentstate` is the strongest lead this investigation has had —
the name describes exactly the kind of data being searched for (live,
device-reported, read-only state), and its existence cleanly explains
every prior negative result without requiring any of them to have been
a mistake.

`roombapy-prime-verify-named-shadows` and `roombapy-prime-validate`
both check all four automatically now. No new mechanics needed —
`get_shadow()`/`get_named_shadow()` already accept any name.

## A third, distinct "find my robot" candidate

Two previous attempts at fixing the known locate/chime bug — a REST
endpoint, then a shadow write — were both tried against a real device
and confirmed not working. A third candidate this time uses a
completely different transport: `send_simple_command("find")`, the
same channel already confirmed working for start/stop/pause/resume/dock.
`"find"` has actually been sitting in this library's own confirmed
command enum for a while, just never tried live. Untested as of this
writing.

## A locate hypothesis, ruled out

`trigger_echo_via_shadow()`, introduced as an experimental locate
mechanism last release, has been tested against a real device and
disproven: the write succeeded (a genuine, non-no-op shadow delta came
back), but the robot did not chime. Kept as a working example of the
shadow-write mechanism itself, not as a locate trigger.

450/450 tests green, ruff clean.
