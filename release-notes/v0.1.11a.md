# v0.1.11a — First public release

This is the first tagged release of `roombapy-prime`, an async Python
client library for iRobot's V4/Prime-generation cloud protocol
(Roomba/Braava devices using the newer `com.irobot.home.prime` app,
distinct from the older Classic-protocol devices `roombapy` already
covers).

**Status: alpha.** Live-confirmed against two independent accounts so
far (different households, same robot model — a Roomba 405 Combo).
Mission control (start/stop/pause/resume/dock) and the large majority
of REST read endpoints are confirmed working against real devices. Map
editing (renaming rooms, etc.) remains unverified against a live
device. See the [README](../README.md#confidence--known-gaps) for the
full, per-feature confidence breakdown, and
[`docs/internal/PRIME_APP_GAP_ANALYSIS_2026-07-11.md`](../docs/internal/PRIME_APP_GAP_ANALYSIS_2026-07-11.md)
for the complete reverse-engineering trail behind every model in this
library.

## Highlights

- **Mission control confirmed live**: `send_simple_command()`
  (start/stop/pause/resume/dock) — watched and confirmed by a real
  user against a real device actually reacting, not just an
  error-free response. This resolved the single biggest open question
  this project had for many sessions.
- **Broad REST coverage**: login, MQTT connection, shadow state/
  settings, favorites (including a confirmed create/verify/delete
  round trip), mission history, map bundle download and parsing,
  schedules, DND settings, cleaning profiles, default routines, robot
  parts/serial data, and more.
- **Full map-bundle content modeling**: all confirmed GeoJSON content
  types (rooms, borders, hazards, furniture, trajectories, coverage,
  dock position, floor plan/type, policy zones, clean zones) with a
  complete, confirmed manifest structure.
- **Security hardening**: URL path segments are properly encoded
  everywhere an identifier is embedded in a request path; credential
  fields (AWS keys, MQTT tokens, per-device certificates) are excluded
  from default object representations and from `--dump-config`'s
  redaction.
- **`models.py` split into a `models/` package**, organized by feature
  area, for maintainability — fully backward compatible, every import
  path unchanged.
- **Three real crashes fixed**, found via a real account's
  `--dump-config` output: `get_default_routines()` would raise an
  `AttributeError` for any account with real routine/scheduling
  history, due to two fields being modeled as lists when the real
  server response is actually a dict keyed by ID. A write-side bug in
  `ScheduleOptions.to_json()` (a missing envelope wrapper) was also
  found and fixed in the same pass.
- **Diagnostic tooling**: `roombapy-prime-validate` (read-only account
  validation + optional `--dump-config` field dump),
  `roombapy-prime-verify-commands` (opt-in, confirmation-gated mission
  command testing against a real device), `roombapy-prime-verify-map-edit`
  (opt-in, narrowly-scoped map-edit testing).

## Known limitations

- Map editing (`edit_map()`) has never been confirmed against a live
  server — the request envelope is bytecode-confirmed, but the
  server's actual behavior is unverified.
- Region-based mission commands (running a favorite, cleaning a
  specific room via `send_routine_command_via_cmd_topic()`) are an
  experimental, unconfirmed hypothesis — use `send_simple_command()`
  for confirmed basic control.
- Only one robot model (Roomba 405 Combo, SKU G185020) has been
  tested live so far, across two different accounts.
- Whether a `RobotStatusV2` structure (battery/charging/dock status)
  appears anywhere in `get_state()`'s response remains unresolved —
  two independent real captures, both idle-robot snapshots, show no
  trace of it.

## Installation

```bash
pip install "roombapy-prime @ git+https://github.com/johnnyh1975/roombapy-prime.git@v0.1.11a"
```

start example.
