# v0.1.11a1

Small, tooling-only follow-up to [v0.1.11a](v0.1.11a.md) — no changes to the
library itself, only to `verify_mission_commands.py`. First release under
this project's revised versioning convention: the alpha counter (`aN`) now
increments for changes like this one, and the third version number is
reserved for larger feature additions going forward.

## What changed

- **`verify_mission_commands.py` gained an interactive mid-mission state
  capture**, inserted between Start and Stop in the core test flow. The
  script's existing before/after snapshots around each command are taken
  only ~3 seconds apart — enough to prove a command was accepted, not
  enough to represent a genuinely active mission. This new step waits for
  explicit confirmation ("robot is now visibly, actively cleaning") before
  calling `get_state()` again — no fixed sleep involved.
- **New `_diff_reported_keys()` helper**, printed directly in the terminal
  (not just written to `--dump-config`): shows which top-level `reported`
  keys are new, missing, or changed in value versus the pre-mission
  baseline.
- 6 new tests, same fully-mocked style as the rest of the test file.
  345/345 tests green, ruff clean.

## Why

Targets the still-open `RobotStatusV2` question directly (see the
"Known limitations" section of [v0.1.11a](v0.1.11a.md) and the README's
"Confidence & known gaps"). Two independent real accounts (chairstacker,
jadestar1864) have so far only ever produced idle-to-idle `get_state()`
captures with identical top-level keys — consistent with either "wrong
data source" or "these fields only populate during an active mission",
and nothing has distinguished between the two yet because no capture has
ever been taken while a robot was confirmed to be actually cleaning. This
release makes that capture possible; it doesn't resolve the question by
itself.

## Known limitations

Unchanged from [v0.1.11a](v0.1.11a.md) — see that note for the full list.
