# v0.1.11a3

Reliability hardening across the board, plus a major structural correction
to map editing — this release bundles everything built since v0.1.11a2:
SSL/certificate clarity across every network layer, a typed exception
hierarchy for translation-key mapping in `ha_roomba_plus`, reconnect-with-
backoff for `watch_state()` (the biggest reliability gap this library
had), and a full correction of all nine V1 map-edit commands prompted by
a live HTTP 500 on a room rename.

## Reliability

- **SSL/certificate error clarity**, extended to every network layer
  (`auth.py`, `rest_client.py`, `mqtt_client.py`) — ported from
  `ha_roomba_plus`'s `cloud_api.py` so every consumer of this library
  (including the standalone `verify-*` scripts) benefits, not just the
  integration.
- **Typed exception subclass hierarchy** (`AuthCredentialsError`,
  `AuthRateLimitedError`, `AuthSSLError`, `AuthConnectionError`,
  `AuthTimeoutError`, and the `Rest*`/`Shadow*` equivalents) — every
  subclass IS-A its base, so existing `except AuthError`-style code keeps
  working, while a consumer that needs finer-grained handling (e.g.
  mapping to Home Assistant translation keys) can now catch the specific
  subclass instead of string-matching messages.
- **Reconnect-with-backoff for `watch_state()`.** Previously, a dropped
  MQTT connection left the generator hung forever with zero signal
  anything was wrong. `on_disconnect` is now wired up, and a drop
  triggers automatic reconnection with exponential backoff (1s → 60s
  capped, unbounded retries) — transparent to the caller's `async for`
  loop.

## Map editing — a major structural correction

A live rename attempt (chairstacker) failed with HTTP 500 against the
previously-assumed request shape. Resolving it meant decompiling the
full `EditMapV1Request.java` source, which corrected far more than the
one failing case:

- Every V1 command's inner body is `{"command": "<snake_case>", "params":
  {...}}`, not the flat `{"type": "<PascalCase>", ...}` previously
  assumed — several discriminator strings differ from what the class
  names suggest (`MergeRooms` → `arrange_room`, `SetVirtualWalls` →
  singular `set_virtual_wall`, `DeletePermanentAreas` → abbreviated
  `del_permanent_area`).
- Three commands (`VirtualWall`, `PermanentArea`, `Furniture`) turned out
  to have custom serializers emitting **positional arrays**, not JSON
  objects — including resolving `VirtualWall`'s long-open
  Linear/Rectangle/NoMopZone discriminator question (a positional int,
  not a `"type"` string).
- **`SetRoomMetadataV1`**, the room-rename/room-category path the app
  actually uses now (`RenameRoomV1`/`SetRoomTypeV1` are app-deprecated,
  though kept available), is fully resolved down to its own serializer
  call: `room_metadata` has exactly two optional keys, `name` and `type`
  — the latter using a new `RoomCategory` enum with snake_case wire
  values, confirmed via the actual serializer logic rather than the more
  plausible-looking (and wrong) camelCase `raw` field the underlying
  Kotlin enum also exposes.
- A real mistake was caught and fixed before ever shipping: an
  intermediate draft conflated `RoomType` with the unrelated `RegionType`
  enum purely because of a similarly-named read-side field. Caught by
  checking the enum definitions directly, not via a live failure.

387/387 tests green, ruff clean.

## Known limitations

Unchanged from [v0.1.11a](v0.1.11a.md) for everything not covered above.
The map-edit corrections are decompilation-confirmed, not yet
live-server-confirmed — `verify_map_edit.py` (now testing
`SetRoomMetadataV1`) will be the first live test of the corrected
structure. The `mqtt_client.py` SSL/connection-error paths and the
reconnect logic are similarly not yet exercised against a real, sustained
failure on live hardware.
