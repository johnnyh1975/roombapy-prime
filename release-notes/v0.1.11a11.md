# v0.1.11a11

The biggest release of this whole investigation so far — the battery
status search this project has been running for weeks is genuinely
resolved, and a real, significant wire-format bug was found and fixed
along the way.

## The battery status search is resolved

A field tester ran the updated named-shadow checker and found four
shadows this project never knew existed — one of them,
`ro-currentstate`, reports `batPct` (battery percentage) live,
alongside `dock`, `cleanMissionStatus`, and eight more keys. This is
what the whole investigation was searching for. Only the key names
are confirmed so far — a follow-up with actual values is still
needed to type this properly, but the *location* of this data is no
longer a mystery.

New models: `CurrentStateShadow`, `StatsShadow`, `ServicesShadow`,
`ConfigInfoShadow` for all four newly-discovered shadows.

## A real, significant wire-format bug fixed: 21 wrong keys across three classes

Investigating region-aware cleaning surfaced something bigger than
expected: `CommandParams` — the parameter structure inside every
region of a region-aware command — had 18 of 39 wire keys wrong, not
just differently-cased. `kotlinx.serialization` silently drops
undeclared keys rather than erroring, meaning any command built with
the old keys would have arrived with these parameters simply missing
— cleaning strength, mop mode, pass count, and more, discarded while
the command itself still went through.

`PolygonEvent` (4 keys) and `CleaningProfile` (1 key, the more
consequential of the two — `command_params` had been silently `None`
against *every* real response) were also found wrong and fixed. A
systematic audit of every remaining unconfirmed field in this
package is now complete, not just sampled.

## Types confirmed for the battery data, without needing a live device

A parallel investigation found that `core::MissionData`'s getters
have bytecode-readable return types — `getBatteryLevelPercentage()`
is a plain `short`, `getResolvedMissionStatus()` and `getDockState()`
are enums (49 and 86 values respectively). A significant correction
came with this, too: `MissionData` has 27 getters, not 7, and doesn't
map 1:1 onto `ro-currentstate`'s keys — it's a larger, aggregated
object. The confirmed types are still directly useful; the mapping
to specific shadow keys remains a hypothesis pending real values.

## Also in this release

- A real, currently-existing redaction gap fixed (`passwordHash`
  wasn't being redacted in diagnostic dumps).
- A cross-reference to `ha_roomba_plus`'s own Classic-tier
  documentation, previously unused, now informing several field
  interpretations with real, already-confirmed behavior from the
  sibling product line.
- `send_routine_command_via_cmd_topic()`'s guidance corrected: an
  earlier recommendation to build commands with just a `favorite_id`
  was backwards — the real app always sends the full resolved
  regions alongside it.

466/466 tests green, ruff clean.
