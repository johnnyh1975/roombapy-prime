# v0.1.11a4

A genuinely new discovery: where live mission status actually lives,
found via native decompilation after a live test ruled out where it
doesn't.

## The finding

A live idle-vs-mid-mission diff of `get_state()` (thank you,
chairstacker) showed the classic shadow's reported state is
byte-identical whether the robot is idle or actively cleaning. Live
mission status does not flow through `get_state()`/`watch_state()` at
all — a hypothesis this project held for a while, now directly disproven
by a real capture.

Following up with native decompilation of `libcorebase.so` found the
actual channel: `{irbt_topic_prefix}/things/{blid}/mission/timeline/report`
(and its `/request` counterpart), built by the same factory
(`AssetIotTopicFactory`) and constructor as the already-live-confirmed
command topic. That shared origin is why we believe the same
`irbt_topic_prefix` applies here too — though that specific application,
and the payload shape on this new topic, are not yet independently
live-confirmed.

## What's new

- `mission_timeline_topic()` (`mqtt_client.py`) and `watch_mission_timeline()`
  (`prime_robot.py`) — subscribes to the new topic with the same
  reconnect-with-backoff behavior `watch_state()` already has.
- `rejected_report_topic()` / `watch_rejected_commands()` — a sibling
  topic found in the same decompilation pass, directly complementing
  the already-live-confirmed `send_simple_command()`: if a command
  appears to succeed but the robot doesn't react, this is where a
  rejection reason would be expected to arrive.
- `watch_raw_topic()` — a thin public wrapper for subscribing to any topic
  this library doesn't have a dedicated method for yet (useful for a
  wildcard subscription, or whatever comes up next).
- `watch_state()`'s reconnect core is now shared code
  (`_watch_topic()`), not duplicated across four methods.
- **New script**: `roombapy-prime-verify-mission-timeline` — purely
  passive by default (never sends anything, watches both new topics at
  once during a mission you start any way you like). Add
  `--start-mission` to have it send the actual start/stop itself
  instead — one script, one terminal, instead of coordinating two.

## A bug caught during the refactor

Extracting the shared reconnect core surfaced a real bug: wrapping an
inner async generator with a bare `async for` doesn't guarantee its
`.aclose()` runs when the outer one is closed. The existing test suite
caught it immediately after the refactor — fixed with
`contextlib.aclosing()`, not shipped broken.

25 new tests since v0.1.11a3. 412/412 tests green, ruff clean.

## Known limitations

Everything about the new mission-timeline topic is exploratory: topic
existence is confirmed from native symbols, but the `irbt_topic_prefix`
assumption and the entire payload shape are unconfirmed until a live
capture exists. That's exactly what `verify-mission-timeline` is for.

Two related leads were also investigated and documented rather than left
for someone else to re-discover: robot position/pose data is real (an
AWS IoT path for it exists) but its topic is built dynamically at
runtime, not from a static literal — no code change follows from this
yet, but `--watch-wildcard` is now the documented, recommended way to
try catching it live. Separately, a mentioned `GetAssetMissionStatusCommand`
read command turned out to be local-HTTPS-only (a legacy "UMI" protocol),
confirmed unusable for this library and not pursued further.

A follow-up pass then found the actual request payload for a position/
pose query as a literal string: `{"do": "get", "args": ["pose"], "id":
<n>}` — explaining why no topic literal exists (the intent lives in the
payload, not the topic). New `send_umi_get_request()` sends this on the
already-confirmed `cmd` topic — EXPERIMENTAL and UNCONFIRMED for this
specific (AWS IoT) transport, since the format is known to also have a
local-only variant. `--try-pose-request` lets a tester try this live,
with its own explicit confirmation prompt.
