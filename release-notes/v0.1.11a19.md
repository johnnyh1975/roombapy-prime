# v0.1.11a19

A dense session: a real, previously-unrecognized shadow modeled for the first time (with genuine
per-device hardware-capability data), a real bug fix based on the first actual field values ever
seen for `ro-stats`, and a new staged test package for settings writes.

## ✨ Added

- **`get_state()`'s classic/unnamed shadow now has a typed model** (`ClassicShadowState`,
  `CapabilityFlags`, `DigiCap`) instead of returning an untyped `ShadowResponse`. Confirmed via a
  real live capture (chairstacker) that first surfaced this shadow's content at all.
  `CapabilityFlags` (36 fields, from the shadow's `cap` object) is the only per-device
  capability data found anywhere in this project so far. Also confirmed: this shadow's own
  `schedHold` is a **separate** value from `rw-settings.sched_hold`, updated independently —
  worth resolving which one the schedule executor actually reads before building anything that
  assumes they're the same value.
- **New `roombapy-prime-verify-settings-write`**, a staged test package for `set_setting()`
  writes against five previously-unverified-effect settings (`child_lock`, `eco_charge`,
  `sched_hold`, `no_auto_passes`, `vac_high`). `--list-settings` (read-only) shows current
  values plus a cross-check between the two `schedHold` sources above. `--toggle KEY` flips one
  setting and reads back whether the write stuck — explicitly does not (and cannot) confirm the
  robot's actual physical behavior changed.
- **`roombapy-prime-verify-mission-timeline` gained `--watch-rrtp-candidate`**, a new, purely
  passive option subscribing to one specific candidate topic for live position/pose data,
  found via native decompilation of `createRobotPositionTopic()`. Kept as a diagnostic-script-
  only candidate until live-confirmed.

## 🐛 Fixed

- **`roombapy-prime-verify-region-commands`' stage 1b (`--send-with-initiator`) now adds
  `initiator="rmtApp"` instead of `"localApp"`.** The old default was borrowed from
  `send_simple_command()`'s own default — itself Classic's literal observed value for a local-
  MQTT connection, never independently confirmed on a real Prime device. A real capture
  (chairstacker) shows Prime's own `rw-software.lastCommand.initiator` as `"rmtApp"` for an
  app-triggered command — the first actual evidence of what this field looks like on Prime.
- **`roombapy-prime-verify-map-edit` now sources room data exclusively from the downloaded map
  bundle, never from `get_active_map_versions()`.** A full APK decompilation confirmed the app
  itself never reads room names from that endpoint at any level of richness — its actual REST
  response class, `P2MapData`, declares no room-metadata field at all. Room-picking now reads
  `RoomFeature.properties.name`/`.room_type` from the bundle instead, guarding the category
  picker against a `room_type` value that doesn't parse as a known `RoomCategory` rather than
  guessing.

## 📄 Docs

- **`rw-software`/`ro-configinfo`'s persistent fetch failures (even after the a14 reconnect fix)
  confirmed as NOT client-side.** `NamedThingShadowTopicFactory::getSupportedPaths()` was
  decompiled and found to do simple regex prefix-matching only, with no per-shadow special
  handling anywhere — ruling out an app-side capability gate. The cause is now understood to be
  server-side (most plausibly timing/provisioning), not something further decompilation can
  resolve.

## Real values, not just field names, confirmed for `ro-stats` this session

A real live capture (chairstacker) finally reached this shadow with actual data, not just key
names. Lifetime mission-outcome counters (`nMssn`/`nMssnC`/`nMssnF`/`nMssnOk`) sum exactly
(25+4+247=276) and cross-validate against `ro-currentstate`'s own mission count from the same
capture. System-uptime hours (7354) are plausible against the device's own registration date.
`unprocessedError` turned out to be a plain string, not an object — real value seen:
`"picea unknown fault code:2105"`.

515/515 tests passing, ruff clean.
