# v0.1.11a5

Real live data came back, and it validated more than expected — plus one
important bug found by looking closely at why part of that data looked
wrong.

## The big one: live mission data, and a schema match nobody expected

chairstacker ran `verify_mission_timeline.py --start-mission` during a
real, active cleaning cycle. Four real events came back: `start` →
`reloc` → `travel` → `room`, each with map IDs, room IDs, area, and
timestamps. New `MissionTimelineReport` model captures the full message
shape.

The genuinely interesting part: the individual event types
(`RoomEvent`, `TravelEvent`, `TentativeLocationEvent`) already existed in
this library — confirmed via static analysis, for a *different* endpoint
(`get_mission_history()`'s historical timeline). They needed **zero**
corrections to parse this live data. The live push channel and the
historical pull endpoint evidently share one event schema — a
cross-confirmation neither investigation alone could have established.

## A real bug, found by looking closely at data that looked slightly off

The same capture showed each event arriving twice, while a wildcard
subscription running at the same time showed nothing at all — despite
clearly having matching traffic to show. That mismatch led to finding a
genuine bug: `_on_message()` matched persistent subscribers with an exact
dictionary lookup on the incoming topic. A wildcard registration's key is
the *pattern* string (e.g. `"…/#"`), which an incoming message's actual
topic can never equal literally. **A wildcard subscription could never
receive anything, in any run, ever** — fixed by matching every
registered pattern properly instead of doing an exact-key lookup.

## Also fixed

- `--start-mission`'s cleanup only sent `stop`, leaving the robot
  stranded wherever it was — chairstacker had to walk over and press the
  button. Now sends `stop` then `dock`.
- A second, related bug found while designing a way to actually test for
  docking events: the watch tasks were cancelled *before* stop/dock were
  sent, meaning any resulting events could never have been captured
  either way. Watching now continues through the whole
  stop → dock → post-dock window (new `--post-dock-watch-seconds`,
  default 30).
- Room renaming (`SetRoomMetadataV1`) is now **live-confirmed**, not just
  decompilation-confirmed — chairstacker renamed a real room and reverted
  it, both confirmed in the app.
- A new map-bundle content type, `policyZones`, confirmed from a second
  real bundle.

5 new tests, including two built from the actual captured mission data
verbatim. 417/417 tests green, ruff clean.

## Known limitations

The mission-timeline schema is confirmed for the four event types
actually observed in one ~25-second capture (start/reloc/travel/room) —
this does not confirm the other confirmed-elsewhere event types (pause,
error, evac, etc.) arrive on this same live channel, only that they're
structurally compatible if they do. `nMssn`'s meaning is still not
confirmed (a lifetime mission counter is a plausible guess). The other
eight V1 map-edit command types remain untested live — only
`SetRoomMetadataV1` (rename) has a real confirmation so far.
