# v0.1.11a14

## Fixed: a real, documented gap in shadow-request reliability

A field report showed a striking pattern: running `verify_named_shadows.py` against the same
device, some shadow queries succeed and every one after that fails — but the exact number
varying between runs (1, then 0, then 3 succeeding across different attempts). That ruled out a
fixed-count explanation (like throttling) on its own.

Investigating this found a real, documented AWS IoT MQTT SDK behavior: after a connection drops
and reconnects, the broker can forget prior subscriptions, and a client that doesn't proactively
reconnect before its next operation will simply never receive a response — silently. A field
report on a completely unrelated project ([aws-iot-device-sdk-js-v2#117](https://github.com/aws/aws-iot-device-sdk-js-v2/issues/117))
describes this exact symptom for shadow topics specifically.

Our own `get_shadow()`/`update_shadow()` had no defense against this for callers doing a plain
sequential series of queries with no reconnect logic of their own (like
`verify_named_shadows.py` — unlike `watch_state()`'s own hardened reconnect loop). After a single
silent mid-run disconnect, every subsequent shadow query in that run would keep trying to
subscribe/publish on a dead connection and time out.

**Fixed**: both methods now check the connection state first and reconnect proactively if it's
known to be down, before attempting the shadow request. Cheap when already connected.

Also added: `verify_named_shadows.py --delay-seconds N` — a cheap, no-downside option to try
alongside this fix, since it's not yet confirmed how much each factor contributes.

470/470 tests green, ruff clean.
