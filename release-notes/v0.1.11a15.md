# v0.1.11a15

The biggest single release this project has had — a resolved multi-session investigation
(locate), four new staged test packages for previously-untested write paths, two enums fully
completed from partial, and a real bug found and fixed along the way.

## 🎉 Resolved: Locate ("find my robot") — CONFIRMED WORKING

`send_simple_command("find")` produces a genuine, audible chime with no robot movement (jayjay,
real device test). This closes out a multi-session search: two earlier mechanisms (a REST
endpoint, a shadow write) were tried first and confirmed **not working**. This is the same
cmd-topic transport already confirmed for start/pause/stop/resume/dock — just a different verb.

## ✨ New: staged test packages for four previously-untested write paths

Each follows the same philosophy: the safest possible test first (resend existing data
unchanged), confirmation gates before anything real gets sent, and no code that leaves a stray
object behind without a clear way to recover.

- **`roombapy-prime-verify-region-commands`** — region-aware mission commands
  (`send_routine_command_via_cmd_topic()`), the riskiest write path this library has. Four
  stages: resend an existing favorite's command unchanged; change one benign field (suction
  level); a from-scratch command for a real room; a hand-built ad-hoc zone (highest risk, needs
  a third safety flag and real, user-supplied geometry).
- **`roombapy-prime-verify-schedule-write`** — schedule writes. Stage 1 resends a household's
  schedules unchanged; stage 2 disables one schedule (the only modification implemented,
  chosen because it can only *prevent* future activity, never cause it).
- **`roombapy-prime-verify-favorite-write`** — favorite writes. Resend unchanged; change only a
  favorite's color (purely cosmetic); a self-cleaning create-then-delete test.
- **`roombapy-prime-verify-virtual-wall-write`** — virtual walls / keep-out / no-mop zones. A
  field report initially suggested add/delta semantics; direct confirmation from the real app's
  own `deleteVirtualWall()` settled this as REPLACE semantics instead, making the same
  "resend unchanged" approach safe here too.

None of these four are live-tested yet — reasoned, safety-checked hypotheses, ready for whoever
wants to try them.

## 🐛 Fixed: a real parsing bug, found via this same research

`PolicyZoneFeature.from_json()` always assumed `Polygon` geometry — would have silently
mis-parsed any real virtual-wall feature (confirmed `LineString` geometry) as if it were a
list-of-rings Polygon. Now dispatches on the GeoJSON object's own `"type"` key. Also resolved the
complete categorization rule for this feature family: there's no separate `"VirtualWall"` type
string at all — a virtual wall is a `"KeepOutZone"`-typed feature whose geometry happens to be a
`LineString` instead of a `Polygon`. This also answers `CommandPolygon.poly`'s previously
unconfirmed coordinate system: confirmed to pass through unchanged at every stage.

## ✨ Two enums fully completed

- **`DockState`** — all 86 values, from parallel native-analysis extraction. Previously only
  discussed in prose, never a real enum. Directly confirms `DockStatus`'s own real captured
  values: `state=301` → `DOCK_READY`, `pw_state=601` → `PAD_WASH_OKAY`, `pd_state=701` →
  `PAD_DRY_OKAY`.
- **`ResolvedMissionStatus`** — all 49 values (0-48), up from ~12 named before.

## Also in this release

- `watch_named_shadows_updates()` — watches `update/accepted` across all named shadows via a
  confirmed-safe single-level (`+`) wildcard, for read-only shadow content `update/delta`
  structurally can never deliver.
- `ConnectionStatusShadow`/`SoftwareStatusShadow` fields properly typed (bool/str/int) from
  decompiled constructor signatures, replacing untyped placeholders.
- `get_time_estimates()`'s body shape partially clarified.
- Reconnect fix narrowed to only relogin when a token is actually near expiry, not on every
  reconnect (see CHANGELOG for the self-correction this addresses).
- `SuctionLevel` enum (5 confirmed values: Invalid/Low/Medium/High/Turbo — no "Auto"). Floor-type
  adaptation isn't a suction-level concept at all — it's the separate `carpet_boost` bool, a
  real, sensor-driven "boost suction when carpet detected" feature (confirmed via iRobot's own
  public product documentation). `CarpetBoostSettings`, a three-way enum found alongside it, is
  confirmed **dead code** in the real app — kept only as a documented dead end.
- `set_setting()`'s docstring clarified with a concrete `carpetBoost` example: the write
  mechanism itself is confirmed to work, but whether any individual key's write actually changes
  the robot's real behavior remains a separate, per-key question.
- Documented a real, structural parity gap: the real app's own "Start" button explicitly fetches
  and sends the account's active cleaning preferences, something `send_simple_command()`
  structurally cannot do — a plausible explanation for a mission always running at unexpectedly
  high power with no adaptation.
