## 1. p2maps REST-Kommandovokabular — vollständig auf Java-Ebene bestätigt

Gefunden unter `com.irobot.irobotdata.maps.internal.p2maps.editing.*`
und `com.irobot.irobotdata.maps.requests.*`. Dies ist eine deutliche
Erhöhung gegenüber der vorherigen Einschätzung im Handoff
("rest_client.py — Niedrig — nur Capability-Flags").

**Einordnung:** Dies ist Java-Quellcode, keine Klassennamen-Vermutung.
Bleibt aber offen: (a) ob dieselbe API 1:1 für V4-BLIDs funktioniert

Beweis für Geräte-Kompatibilität), (b) die Domain/Basis-URL wurde nie
explizit gefunden, nur relative Pfade.

## 2. Live-Karte & Live-Position — neuer, unabhängiger Mechanismus gefunden

Dies berührt direkt Abschnitt 3 des Handoffs ("Live-Pose über die
Cloud — bisher stärkstes, aber unbestätigtes Indiz").

**Bisheriger Stand (nativ, `liblegacyCore.so`):** Vermutung eines
AWS-IoT-Shadow-basierten Wegs (`GetRobotPositionCommand`,
`ThingShadowConstants::kRobotPositionSuffixTemplate`) — Command-Muster,
kein kontinuierlicher Push.

**Neuer Fund (Kotlin, `com.irobot.irobotdata.maps.requests.livemap`):**
Ein komplett anderer, REST+MQTT-Hybrid-Mechanismus:

1. `GET /v1/p2maps/livemap?robotId={blid}` →
   ```json
   {"mqtt_topic": "...", "livemap_url": "..." (optional)}
   ```
2. Das zurückgegebene `mqtt_topic` wird auf der **bereits bestehenden**
   AWS-IoT-WebSocket-Verbindung abonniert (derselbe Custom-Authorizer-
   Pfad, kein neuer Verbindungsaufbau).
3. Nachrichten auf diesem Topic, zwei Formen:
   ```json
   {"timestamp": "...", "map_update": {"livemap_url": "..."}}
   ```
   ```json
   {"timestamp": "...", "update_expire_ts": "...", "pos_update": {...}}
   ```
4. `pos_update.cur_path` ist ein **flaches JSON-Array**:
   `[seq_nr, x1,y1,orient1,mode1, x2,y2,orient2,mode2, ..., epoch_ts]`
   — Länge ist `(2 + 4*n)` für `n` Positionspunkte (im Code über
   `(size - 2) % 4 == 0` validiert). Orientierung wird beim Parsen
   um `+π` verschoben (Konvention ungeklärt).

**Das bestätigt exakt** die native Vermutung aus
`RobotPositionEventImpl` (Vektor von `PositionData` pro Ereignis,
trajektorienartig) — nur eben über einen anderen Transportweg als
ursprünglich vermutet.

**Offene Frage, bewusst nicht selbst entschieden:** Koexistieren beide
Mechanismen (älterer nativer Shadow-Weg, neuerer App-Kotlin-REST+MQTT-
Weg), oder hat der neue Fund den alten schlicht abgelöst? Nicht
gegeneinander getestet. Für Aders V4-Arbeit ist der neue Fund
vermutlich der relevantere, weil er der tatsächlich von der aktuellen
App genutzte Weg ist (nicht nur eine native Capability-Möglichkeit).
